# `ud363.php`

HTTP upload/download entrypoint; the **module name is a placeholder** referencing XMPP **[XEP-0363: HTTP File Upload](https://xmpp.org/extensions/xep-0363.html)** by number — **not** a specific “UD363 hardware device”.

## Purpose

Server-side HTTP surface for **uploading and downloading files** (same scenario XEP-0363 targets: grant upload permission, then expose a stable download URL).

`server/ud363.php` may be minimal in-tree; the intended surface area is [`server/api/ud363/`](../api/ud363/README.md) plus backend [`server/backends/ud363/`](../backends/ud363/README.md).

## Normative source: XEP-0363

| Field | Value |
|------|--------|
| Title | HTTP File Upload |
| Status | Stable (this summary uses **v1.2.0**, 2025-06-17) |
| Idea | Client asks an upload service for a **slot**: paired HTTPS URLs for **PUT** (upload) and **GET** (download). |

In the canonical protocol, the exchange is **XMPP IQ** stanzas (`<request xmlns='urn:xmpp:http:upload:0' …>` → `<slot><put/><get/></slot>`). The RBT **`ud363`** module is **not** a full XMPP implementation; it is an **HTTP-shaped stub** with the same intent — see below.

## Reference notes: XEP-0363 (what we reuse by meaning)

Full text: [XEP-0363](https://xmpp.org/extensions/xep-0363.html).

XEP-0363 describes an HTTP file upload flow where a client first requests a **slot** and then uploads a file to the returned PUT URL.

- **Slot model**: a tuple of **PUT URL** (upload) + **GET URL** (download).
- **Slot request fields**: `filename`, `size` (bytes), optional `content-type`.
- **Allowed PUT headers** (if the slot response provides them): only `Authorization`, `Cookie`, `Expires`.
  - Treat headers as **opaque**; strip newlines; preserve ordering; ignore any other header names.
- **Upload requirements**:
  - reject upload if `Content-Length` mismatches the requested slot `size`.
  - reject if `Content-Type` was specified and doesn’t match (recommended).
  - HTTP `201` indicates the file is available via the GET URL.
  - PUT URLs should have a short TTL (XEP suggests ~300s).
- **Upload purpose / retention idea**: buckets like `message` (default), `profile`, `ephemeral (expire-before)`, `permanent`.
- **Error patterns** to expose to clients: file-too-large (`max-file-size`), temporary quota errors (“retry at”), forbidden/not allowed.
- **Web clients**: CORS considerations for `OPTIONS/HEAD/GET/PUT`, `Authorization`, `Content-Type`, and possibly credentials.
- **Security**:
  - isolate the download domain; use strict CSP (e.g. `default-src 'none'; frame-ancestors 'none';`).
  - use long randomized URLs (non-guessable).
  - note that upload/download leaks client IP; and files may be stored unencrypted without E2EE.

## Discovery (for XEP completeness)

In XMPP, clients locate the upload component via **Service Discovery**: feature **`urn:xmpp:http:upload:0`**, optional **`max-file-size`** (bytes) via `jabber:x:data`. The HTTP API here does not mirror disco automatically — document limits and PUT TTL explicitly when implementing slots.

## Naming note

The **`ud363`** prefix is a mnemonic for **0363**, not vendor hardware. The stack remains a **thin stub** until slots, storage (`files`, etc.), and retention policies are wired end-to-end.
