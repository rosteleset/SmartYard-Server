# `ud363.php`

HTTP endpoint для upload/download; **имя модуля — заготовка**: оно отсылает к номеру расширения XMPP **[XEP-0363: HTTP File Upload](https://xmpp.org/extensions/xep-0363.html)** (не к конкретному «железу UD363»).

## Назначение

Серверная HTTP-точка входа для **загрузки и скачивания файлов** (по замыслу — тот же сценарий, что покрывает XEP-0363: выдать клиенту возможность залить файл на HTTP и получить стабильную ссылку на скачивание).

Файл `server/ud363.php` в репозитории может быть минимальным; основная проработка API идёт через [`server/api/ud363/`](../api/ud363/README.ru.md) и backend [`server/backends/ud363/`](../backends/ud363/README.ru.md).

## Нормативный источник: XEP-0363

| Поле | Значение |
|------|-----------|
| Название | HTTP File Upload |
| Статус | Stable (на момент конспекта — версия **1.2.0**, 2025-06-17) |
| Идея | Клиент через XMPP запрашивает у сервиса загрузки **разрешение** и получает пару HTTPS URL (**slot**): куда **PUT**’ить файл и откуда потом **GET**. |

В «настоящем» XEP обмен идёт **XMPP IQ** (`<request xmlns='urn:xmpp:http:upload:0' …>` → `<slot><put/><get/></slot>`). В этом проекте модуль **`ud363`** — это **не** полный парсер XMPP: это **HTTP-слой с тем же смыслом** (слот → загрузка → выдача URL), см. раздел ниже.

## Конспект: XEP-0363 (что используем по смыслу)

Полный текст: [XEP-0363](https://xmpp.org/extensions/xep-0363.html).

XEP-0363 описывает схему HTTP upload, где клиент сначала запрашивает **slot**, а затем грузит файл по PUT URL.

- **Slot-модель**: пара URL — **PUT URL** (загрузка) + **GET URL** (скачивание).
- **Параметры запроса слота**: `filename`, `size` (в байтах), опционально `content-type`.
- **Разрешённые заголовки для PUT** (если их вернул slot): только `Authorization`, `Cookie`, `Expires`.
  - Заголовки считать **opaque**; вычищать newline; сохранять порядок; игнорировать остальные имена заголовков.
- **Требования к загрузке**:
  - отклонять upload, если `Content-Length` не совпадает с `size` из слота.
  - отклонять, если `Content-Type` был указан заранее и не совпадает (рекомендуется).
  - HTTP `201` означает, что файл доступен по GET URL.
  - PUT URL должен жить недолго (XEP рекомендует ~300с).
- **Purpose / retention идея**: “бакеты” `message` (по умолчанию), `profile`, `ephemeral (expire-before)`, `permanent`.
- **Паттерны ошибок**, которые полезно отдавать клиенту: file-too-large (`max-file-size`), временные quota ошибки (“retry at”), forbidden/not allowed.
- **Web-клиенты**: CORS для `OPTIONS/HEAD/GET/PUT`, заголовков `Authorization`, `Content-Type`, и при необходимости credentials.
- **Безопасность**:
  - изолировать домен скачивания; строгий CSP (например `default-src 'none'; frame-ancestors 'none';`).
  - длинные рандомные URL (неугадываемые).
  - учитывать, что upload/download раскрывает IP клиента; и без E2EE файлы могут храниться незашифрованными.

## Обнаружение сервиса (для полноты картины XEP)

В самом XMPP клиент находит сервис по **Service Discovery**: фича **`urn:xmpp:http:upload:0`**, опционально **`max-file-size`** (байты) в форме `jabber:x:data`. Для HTTP API RBT это не дублируется автоматически — при реализации своего слота имеет смысл явно документировать лимиты и TTL PUT URL.

## Почему это не «просто номер контроллера»

Префикс **`ud363`** в коде — мнемоника **0363**, чтобы не путать с произвольным оборудованием. Реализация намеренно остаётся **тонкой заготовкой** до полной связки со слотами, хранилищем (`files` и т.д.) и политиками retention.
