# `ud363.php`

Endpoint для upload/download (по смыслу похоже на XEP-0363 “HTTP File Upload”).

## Назначение

Серверная HTTP-точка входа для **загрузки и скачивания файлов**.

Название “363” — отсылка к **XEP-0363** (XMPP HTTP File Upload): реализация **похожа по смыслу**, но не является 1:1 реализацией протокола.

## Конспект: XEP-0363 (что используем по смыслу)

Ссылка: `https://xmpp.org/extensions/xep-0363.html`

XEP-0363 описывает схему HTTP upload, где клиент сначала запрашивает **slot**, а затем грузит файл по PUT URL.

- **Slot-модель**: пара URL — **PUT URL** (загрузка) + **GET URL** (скачивание).
- **Параметры запроса слота**: `filename`, `size` (в байтах), опционально `content-type`.
- **Разрешённые заголовки для PUT** (если их вернул slot): только `Authorization`, `Cookie`, `Expires`.
  - Заголовки считать **opaque**; вычищать newline; сохранять порядок; игнорировать остальные имена заголовков.
- **Требования к загрузке**:
  - отклонять upload, если `Content-Length` не совпадает с `size` из слота.
  - отклонять, если `Content-Type` был указан заранее и не совпадает (рекомендуется).
  - HTTP `201` означает, что файл доступен по GET URL.
  - PUT URL должен жить недолго (XEP рекомендует ~300с).
- **Purpose / retention идея**: “бакеты” `message` (по умолчанию), `profile`, `ephemeral (expire-before)`, `permanent`.
- **Паттерны ошибок**, которые полезно отдавать клиенту: file-too-large (`max-file-size`), временные quota ошибки (“retry at”), forbidden/not allowed.
- **Web-клиенты**: CORS для `OPTIONS/HEAD/GET/PUT`, заголовков `Authorization`, `Content-Type`, и при необходимости credentials.
- **Безопасность**:
  - изолировать домен скачивания; строгий CSP (например `default-src 'none'; frame-ancestors 'none';`).
  - длинные рандомные URL (неугадываемые).
  - учитывать, что upload/download раскрывает IP клиента; и без E2EE файлы могут храниться незашифрованными.
