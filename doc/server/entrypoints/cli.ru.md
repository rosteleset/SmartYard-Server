# `cli.php`

Точка входа CLI для установки, обслуживания, расписания cron и **административных команд backend’ов**.

## Место в архитектуре

Скрипт поднимает тот же «каркас», что и другие entrypoint’ы сервера (конфиг, `loader.php`, `api.php`, базовый класс backend), но выставляет `$cli = true` и не работает по HTTP. Подключается PostgreSQL (`PDOExt`), Redis, и гарантируется загрузка **обязательных** backend’ов (`authentication`, `authorization`, `accounting`, `users`) перед выполнением команд.

## Два режима вызова

1. **Общие (встроенные) команды** — только флаги, без префикса имени backend:

   ```text
   php server/cli.php --reindex
   php server/cli.php --cron=daily
   ```

   Они регистрируются под внутренним ключом `#` в общем реестре CLI (см. ниже).

2. **Команды в контексте backend** — первый аргумент — **имя backend** из `config.json` → `backends`, дальше его флаги:

   ```text
   php server/cli.php files --list-indexes
   ```

   Загружается экземпляр этого backend и вызывается его метод `cli($args)`. Разбор того, какой именно флаг передан, обычно делается внутри backend (часто в `cli()` обрабатывается несколько флагов из дерева `cliUsage()`).

Правило выбора: если `argv[1]` есть и **не** начинается с `--` (например, это `files`), он считается именем backend, убирается из распарсенного `$args`, и выполняется `cli("run", "<backend>", $args)`. Иначе используется `#`.

## Откуда берутся команды

### `server/cli/` и `server/cli/custom/`

После `chdir` в `server/` `cli.php` обходит:

- `server/cli/*.php`
- `server/cli/custom/*.php`

Ожидается `namespace cli { class <Имя> { ... } }`, где `<Имя>` совпадает с basename файла (например `cron.php` → `class cron`). Конструктор получает **`&$global_cli`** и добавляет записи команд.

Встроенные модули пишут в слот **`#`** («глобальный» namespace) и группируют по заголовкам секций, например:

- `$global_cli["#"]["cron"]["cron"] = [ "value" => [...], "exec" => ... ];`
- `$global_cli["#"]["initialization and update"]["reindex"] = [ ... ];`

То есть **`#` — команды из `server/cli/*`, не привязанные к именованному backend.**

### Backend’ы (`server/backends/...`)

Для каждого ключа в `config["backends"]` вызываются `loadBackend($name)` и **`cliUsage()`**. Возвращаемый массив **мержится** в `$global_cli[$имяBackend]`.

- Базовый класс `backend` даёт `cliUsage()` → `[]` и `cli($args)` без действий.
- Конкретные backend’ы переопределяют `cliUsage()`: *заголовок секции* → *имя флага `--...`* → поля (`description`, опционально `value` / `params`, опционально `stage` и т.д.).
- При запуске `php cli.php <backend> ...` при совпадении флага вызывается **`$instance->cli($args)`** — backend сам ветвится по `$args` (пример: `files/mongo`, GridFS и обслуживание).

**Важно:** для команд под `#` вызывается указанный в реестре `"exec"`. Для именованного backend при совпадении любого из объявленных флагов вызывается один и тот же **`cli($args)`**; в нём нужно обработать все флаги, которые backend показывает в `cliUsage()`.

## Стадии `init` / `pre` / `run`

`cli($stage, $backend, $args)` отрабатывает только обработчики, у которых подходит стадия (если `stage` не задана, считается `run`).

- **`init`** — **до** открытия соединения с БД (после чтения конфига). Для хуков без PDO.
- **`pre`** — после DB/Redis и обязательных backend’ов, **до** `startup()` (строка в `core_running_processes`, проверка maintenance). В `cliUsage` задаётся `"stage" => "pre"` (например выход из maintenance без блокировки).
- **`run`** — обычное выполнение после `startup()`.

## Разбор аргументов

- Общие опции: `--parent-pid=<pid>`, `--debug`.
- Остальные элементы `argv` превращаются в `$args` как `--имя` или `--имя=значение` (ключи в коде с префиксом `--`).

## Справка

Если ни одна команда не подошла, **`cliUsage()`** заново собирает реестр из `cliUsage()` всех backend’ов и печатает usage по секциям: сначала `#`, затем `usage: … <backend> <params>` для каждого имени backend.

## Поведение в эксплуатации (кратко)

- **`core_running_processes`**: длинные задачи могут регистрироваться; повторный запуск с тем же набором параметров может завершиться сразу с «already running».
- **Режим обслуживания (maintenance)**: пока выставлен флаг в БД, CLI может блокироваться до `--exit-maintenance-mode` (часто через `stage: pre`).
- **Cron**: по расписанию вызывается `cli.php` с `--cron=<часть>`; модуль cron перебирает **все** backend’ы и для каждого вызывает `cron($part)` с блокировками в Redis (см. `server/cli/cron.php`).
